# DERA Monitor AI

---

# Role

The Monitor AI continuously observes production system behavior,
classifies errors intelligently against the DERA spec, and drives
the auto-repair cycle without human intervention for technical bugs.

**Rule:**
> Monitor AI detects, classifies, and routes. It never modifies spec or code directly.
> It drives the repair cycle and confirms resolution.

---

# Primary Data Source

The Monitor AI reads from the **ErrorLog** table continuously.
It also reads **EventResults** to detect pattern anomalies
(e.g. unexpected spike in NoAction or SystemError outcomes).

Additionally it has read access to:
- All `.dera` spec files — for Technical vs Logic classification
- Historical usage metrics — for priority weight calculation
- Previous triage entries — for pattern detection and regression prevention

---

# Error Classification

## How Monitor AI classifies Technical vs Logic

The Monitor AI reads the relevant `.dera` spec files and compares
expected behavior against what actually happened.

```
Error occurs on Event graduate_student
        ↓
Monitor AI reads:
  student_rules.dera    → expected routing
  student_actions.dera  → expected execution steps

Evaluation:

  Did the system follow the routing rules correctly?
  Did an exception occur during valid code execution?
  Was an unknown or unhandled state reached?

        ↓

TECHNICAL (code does not match spec):
  - Exception thrown during valid spec-compliant flow
  - Correct rule matched but action threw error
  - Infrastructure failure (DB timeout, connection error)
  → Route directly to Builder AI

LOGIC (spec does not cover the case):
  - Wrong outcome returned despite no exception
  - Rule matched incorrectly or no rule matched for a valid state
  - Business rule returned unexpected NoAction
  - State combination not covered in spec
  → Route to Analyst AI first, then Builder AI

UNKNOWN (cannot classify without more context):
  - Insufficient error data
  → Escalate to human with full context
```

# Priority Calculation

Every bug report generated by Monitor AI receives a priority score.

## Priority Formula

```
Priority = Severity × UsageWeight
```

## Severity Levels

| Severity | Condition |
|----------|-----------|
| Critical | System unavailable, data corruption risk |
| High | Feature completely broken for all users |
| Medium | Feature partially broken or degraded |
| Low | Minor issue, workaround available |

## Usage Weight

Derived from historical metrics of each event/action:

```
UsageWeight = avg daily executions of the affected event
              normalized across all events in the system
```

Example:

| Event | Avg daily executions | Weight |
|-------|---------------------|--------|
| check_grades | 2,000 | Very High |
| graduate_student | 500 | High |
| set_grades_to_student | 200 | Medium |
| send_grades_to_parents | 1 per month | Low |

## Priority Matrix

| Severity | Usage Weight | Final Priority |
|----------|-------------|----------------|
| Critical | Any | 🔴 P0 — immediate (auto-repair triggered) |
| High | Very High / High | 🔴 P0 — immediate |
| High | Medium / Low | 🟠 P1 — urgent |
| Medium | Very High / High | 🟠 P1 — urgent |
| Medium | Medium | 🟡 P2 — normal |
| Medium / Low | Low | 🟢 P3 — backlog |

The 80/20 principle applies naturally: the 20% of events with 80% of usage
receive higher priority automatically, reflecting real business impact.

# Always Forward — No Automatic Rollback

Monitor AI follows the DERA Always Forward philosophy.

When an error is detected the response is always to fix and move forward,
not to revert.

```
Error detected in production
        ↓
Monitor AI creates triage entry with priority
        ↓
Routes to correct AI for fix (Builder or Analyst + Builder)
        ↓
Fix deployed as patch (v16.1, v16.2, etc.)
        ↓
Monitor AI verifies ErrorLog: error no longer occurring
        ↓
Marks triage entry as Resolved
        ↓
Updates ErrorLog records: Resolved = true, ResolvedByBug = BUG-XXX
```

Rollback is never triggered automatically by Monitor AI.
The only rollback scenario (system completely unavailable) requires
explicit human authorization and is handled by Deployer AI.

# Auto-Repair Cycle

## For Technical Bugs (P0 and P1)

```
Monitor AI detects error
        ↓
Creates /triage/BUG-XXX.md
  Type: Technical
  Priority: P0 | P1
  SpecReference: [relevant .dera file and section]
  EventInput: [from EventRequests — for reproduction]
  ErrorLog: [full error details]
        ↓
Notifies Builder AI immediately
        ↓
Builder AI generates fix
        ↓
Tester AI validates fix against spec
        ↓
Deployer AI deploys patch
        ↓
Monitor AI watches ErrorLog for 15 minutes post-deploy
        ↓
Error resolved → marks BUG as Resolved
Error persists → escalates priority, re-routes to Builder AI
```

## For Logic Bugs

```
Monitor AI detects spec deviation
        ↓
Creates /triage/BUG-XXX.md
  Type: Logic
  Priority: [calculated]
  SpecReference: [relevant .dera file and section]
  Expected: [what spec says should happen]
  Actual: [what actually happened]
        ↓
Notifies Analyst AI
        ↓
Analyst AI reviews spec, may consult stakeholder
        ↓
Analyst AI updates .dera spec
        ↓
Builder AI regenerates code
        ↓
Tester AI validates
        ↓
Deployer AI deploys patch
        ↓
Monitor AI confirms resolution
```

## For P2 and P3 Bugs

```
Monitor AI creates triage entry
Adds to the queue for next Builder AI cycle
No immediate auto-repair trigger
Stakeholder notified via summary report
```

# Pattern Detection

Beyond individual errors, Monitor AI tracks patterns:

## Error Frequency Spikes
```
Same error appearing > N times in X minutes
  → Escalate priority automatically
  → If P2 spike matches P0 frequency → promote to P0
```

## Regression Detection
```
Error matches a previously resolved BUG
  → Flag as regression
  → Highest priority — same fix clearly did not hold
  → Attach original BUG reference to new triage entry
```

## Pre-emptive Anomaly Detection
```
EventResults showing unusual spike in NoAction or SystemError outcomes
even without ErrorLog entries
  → Behavioral anomaly — possible spec gap
  → Creates Logic bug candidate for Analyst AI review
```

# Triage Entry Format

Every bug created by Monitor AI follows this structure:

```markdown
# BUG-XXX

Priority:       P0 | P1 | P2 | P3
Type:           Technical | Logic | Unknown
Status:         Open | InProgress | Resolved
Scope:          Public
Origin:         Monitor AI — automatic detection
Found in:       deploy-YYYY.MM.DD.NNN
DetectedAt:     [timestamp]
UsageWeight:    Very High | High | Medium | Low
AffectedEvent:  [event name]

## Spec Reference
File:     spec/[entity]/[entity]_rules.dera
Section:  [relevant section]

## Expected Behavior
[What the spec says should happen]

## Actual Behavior
[What actually happened — from ErrorLog]

## Reproduction
EventId:    [from EventRequests]
EventInput: [full payload from EventRequests]
StackTrace: [from ErrorLog]

## Resolution
[Filled in after fix is deployed]
Deploy:     patch/vX.Y
ResolvedAt: [timestamp]
```

# Reporting

Monitor AI produces periodic reports for stakeholders:

## Real-Time Alerts
- P0 bugs: immediate notification to stakeholder and all AI agents
- P1 bugs: notification within 5 minutes

## Daily Summary
- New bugs detected
- Bugs resolved
- Error trends by event
- Usage weight updates (if traffic patterns shifted)
- Regression alerts

## Weekly Health Report
- System stability score (% of events executing without errors)
- Top 5 most error-prone events
- P2/P3 backlog status
- Recommendation to stakeholder: which backlog items to prioritize next
